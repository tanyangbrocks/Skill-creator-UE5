## 五、裝備機制更新

「裝備物品」移除獨立按鍵（舊 Q 鍵 `OnEquipItem`）。裝備途徑只剩兩種：

1. **手動拖曳**：在物品欄（R）中把物品拖曳至裝備欄（T）對應槽位
2. **自動裝備**：當裝備槽為空時，用右鍵或其他指定操作自動裝備物品欄中第一個符合槽位的物品

> 自動裝備的精確觸發點（右鍵物品欄格子？打開裝備欄自動填滿？）待 UI 互動細節設計時決定。

### D-5 碎塊系統（已有設計，尚未實作）

已確認：`origin text setting/dig & place.txt §3` + `docs/plan-mining-placement.md R-5` 有完整設計，以下是摘要。

**設計確認（不需再問）：**

1. **碎塊是實體物件**（非純視覺粒子）：有物理法則，可被踩到，可被玩家撿起與投擲
2. **生命週期**：永久存在，直到玩家撿起（同 DroppedItemManager 掉落物行為）
3. **觸發條件**：外力摧毀（爆炸/斬斷/碾碎/崩塌）產生的碎塊，門檻為 ≥ 1% material unit（≈ 10 tiles）才形成碎塊實體，低於門檻直接消失
4. **與 DroppedItemManager 整合**：同一套系統，但物品 ID 是 `FragmentXxx`（石塊碎片、木材碎片等），而非 `BlockXxx`
5. **撿起後進物品欄**：佔幾 % material unit → 幾個「碎片」（整數，無條件捨去）
6. **100 碎片 → 合成 1 方塊**：屬合成系統範疇，R-5 本身不實作

**換算尺度（`plan-mining-placement.md §八`）：**
- 1 material unit ≈ 1000 tiles（立方體半徑 5）= 100 碎片
- 1% 門檻 = 10 tiles，純整數，無次 tile 精度需求

**UE5 實作狀態（R-5）：⛔ 尚未實作**  
前次 session 曾誤把 Godot 採掘掉落表填成 `FragmentXxx`，已改回 `BlockXxx`（正確）。碎片系統需要的 `DestroyReason` enum、Fragment ItemId 變體、爆炸掉落分流邏輯均未建立。

**UE5 實作步驟（參照 `plan-mining-placement.md R-5`）：**
1. `TileWorld3D.h` 加 `EDestroyReason`（Mining / Explosion / Slash / Crush / Collapse）
2. `ItemId.h` 加 `FragmentDirt/Stone/Sand/Wood/...` 等 Fragment 變體
3. `MaterialRegistry` 各材質加 `FragmentItem` 欄位
4. `DroppedItemManager` 依 DestroyReason 分流：採掘 = 原完整 Block 掉落；爆炸/外力 = 計算 tile 數 ÷ 1000 × 比例 → 掉落碎片數
5. 爆炸/技能命中在外層統計總摧毀 tile 數，集中計算（不能每個 tile 各自掉碎片）

**粒子 vs. 應力算法分離（`plan-mining-placement.md §六`）：**  
三層完全獨立：Tile CA 物理（GPU）、次像素視覺粒子（V 系列）、應力/崩塌判定（tile-group FloodFill，僅在 tile 被摧毀後觸發）。R-5 碎片系統只接應力層輸出，不處理視覺層。

## 七、缺少的核心系統

### S-1：玩家狀態機（PlayerStateMachine）
**影響**：幾乎所有動作

目前 PlayerController 是 flat class，缺乏明確狀態。建議引入：

```
Idle → Running → Crouching
     ↗ Jumping → Falling → Landing
              ↘ Aerial（二段跳、空中技）
     → Sprinting → Sliding
     → Rolling（翻滾，無敵幀）
     → Flying → FastFlying
     → Attacking（Light / Heavy / Charged）→ Stunned
     → Guarding → Parrying
     → Dashing（Forward / Backward）
```

---

### S-2：攻擊框架（AttackFramework）
**影響**：J 輕攻擊、J+U 重攻擊、長按蓄力、所有空中攻擊、交鋒規則

**架構確認（2026-06-23）**：與技能系統完全分離，不走 SpellRunner VM。

需要：
- `AttackType` enum（Light, Heavy, Charged, Aerial, Plunge, Rising, Rush）
- `AttackHitbox`（存活幀數、傷害、命中點）
- `AttackHitAction`（派發到 ActionBus，含 AttackType）
- 碰撞判定：在指定幀數內掃描 Hitbox 範圍內的敵人
- 彈反/鬼影共用的「攻擊即將到來」偵測窗口（6 幀常數）

---

### S-3：鎖敵系統（TargetLockSystem）
**影響**：E 鎖敵、Tab 切換目標

**確認（2026-06-23）**：鎖敵不影響右鍵行為（右鍵仍依 IInteractable 優先判定）。

需要：
- `LockTarget`（nullable Enemy reference）
- 鎖定優先演算法（距離 + 螢幕中心角度）
- 鎖定時相機偏轉追蹤
- 玩家準心 UI 指示器
- 脫戰/死亡時自動解鎖

---

### S-4：防禦/彈反框架（GuardParrySystem）
**影響**：防禦、彈反近戰/遠程、破防

**確認（2026-06-23）**：彈反窗口 = 6 幀（@60fps），以具名常數定義。

需要：
- Guard State（在狀態機 S-1 中）
- Parry Window（進入 Guard 後 N 幀內 = 彈反有效，N = 6 起始值）
- 投射物反射：`EnemyProjectile._dir *= -1`，傷害歸屬改為玩家端

---

### S-5：移速乘數系統（SpeedSystem）
**影響**：疾跑、超速、飛行速度

目前固定速度常數，改為：
`baseSpeed × sprintMult × flightMult × statusMult`

`statusMult` 已有（來自 CharacterStats），其餘需補。

---

### S-6：藥水袋系統（PotionBagSystem）
**影響**：Q 服用藥水、Y 藥水袋面板

資料結構、UI、物品分類細節待 §六 D-4 確認後設計。

---

### S-7：地圖系統（MapSystem）
**影響**：M 地圖

未設計；最小版本 = 小地圖（minimap）顯示已生成 chunks 的上視圖。

---

### S-8：殘影/虛化系統（AfterimageFX）
**影響**：L 後撤殘影、鬼影虛化

殘影 = 後撤路徑上留下半透明複製 mesh，持續約 0.3 秒後消失。  
虛化 = 無敵幀（可用 ActionBus 過濾傷害，類似 `green_invincible`）。
