# 開發血汗錄

> 記錄那些「程式碼看起來都對，但畫面就是不對」的修羅場。技術細節在 commit/issues.md 裡都有，這份檔案記的是過程——花了多久、走錯幾次、最後怎麼抓到的，給未來的人（不管是人類還是 AI）一個警世故事。

---

## 第 1 案：標題畫面的幽靈（2026-06-20 ~ 2026-06-21）

**症狀**：GameFlowWidget 整套重做完，`Build 0 錯誤 0 警告`，commit 完，回報「完成」。使用者按下 Play——直接跳過標題畫面，進到遊戲場景。HP/MP 條都在，就是沒有那層「SkillCreator」標題。

**第一輪診斷（自己跑 `-game`，純看 log）**：

跑了三次 `UnrealEditor-Cmd.exe -game`，log 都乖乖印著：

```
LogTemp: SkillCreatorGameMode: no pending world/character — showing GameFlowWidget
LogTemp: SkillCreatorGameMode: GameFlowWidget created, added to viewport, game paused
```

沒有任何 Error。看起來邏輯都對，於是先去修了個真的存在但無關的小 bug（`GlobalDefaultGameMode` ini 路徑寫錯 `ASkillCreatorGameMode` 多了個 `A`），回報「應該沒問題了」。

**使用者**：「你還是失敗了，進入遊戲依然沒有看到標題畫面。」

——這時候才發現，光看 log 完全不夠。Log 只能證明「程式碼執行到那一行沒有 crash」，不能證明「畫面上真的畫出東西」。

**第二輪診斷（追到變成 stale binary 烏龍）**：

發現使用者測的 `Binaries/Win64/SkillCreatorUE5.exe` 是 4 天前編的，跟 Editor target 是兩個完全獨立的編譯產物，互不更新。修正了，重 build 兩個 target。使用者再測——

**還是沒有畫面。**

**第三輪診斷（瘋狗式排除法，大概燒了 2 小時）**：

接下來進入了一段非常痛苦的「拿掉一個變因、重 build、重測」循環，每輪大概 3~4 分鐘：

| # | 排除的假設 | 結果 |
|---|-----------|------|
| 1 | Widget 尺寸算成 0×0 | 量出來 `AbsSize=(900,650)` 滿版，不是 |
| 2 | `SetAnchorsInViewport`/`SetAlignmentInViewport` 設定有問題 | 拿掉照樣壞，UE5.7 預設本來就是滿版拉伸 |
| 3 | `SetPause`/`SetInputMode`/`SetShowMouseCursor` 副作用 | 拿掉照樣壞 |
| 4 | 跟 `AHUD::BeginPlay()` 的執行順序有關 | 延後 30 個 tick 建立，照樣壞 |
| 5 | Border 沒設 Brush 資源不會畫 | 換成最簡單的純紅色 Border + 文字，照樣壞 |
| 6 | 連 Border 都不要，只留一個黃色文字當 root | **還是完全沒畫出來** |
| 7 | 是不是「誰呼叫 `CreateWidget`」這個 context 的問題 | 搬到跟會渲染的 HUDWidget 同一個 context 建立，照樣壞 |
| 8 | `UCLASS()` 缺 `BlueprintType, Blueprintable` 標記 | 補上，照樣壞 |

每一項都合理懷疑、每一項都查了 UE5.7 引擎原始碼驗證、每一項都實機重測——**全部沒用**。

中途還挖出一個離題的小發現：git log 裡有個 6/20 的 commit，標題正是「修復 GameFlowWidget 在 PIE 中永遠不可見的問題」，用了一個叫 `SetOffsetsInViewport()` 的 API。一查引擎原始碼——**這個方法在 UE 5.7 根本不存在**。後來有個 session 為了修編譯錯誤把那行刪了，但沒人意識到這可能等於把「修復」本身也刪掉了。也沒人重新看過畫面。這個 bug 大概已經潛伏了一天多，從頭到尾沒人真的用眼睛確認過。

**轉折點：Widget Reflector**

純程式碼/log 排查走到死路，唯一剩下的工具是 Slate 的 Widget Reflector——直接看引擎內部真正畫出來的 widget 樹。

這段也不輕鬆：

- 一開始選了「Standalone Game」play 模式——這是**獨立 process**，Widget Reflector 在 Editor process 裡，完全看不到另一個 process 的東西。換成「New Editor Window (PIE)」才解決。
- Widget Reflector 視窗一開始太小，操作幾次後不小心被拖到背景去（使用者一度以為「視窗消失了」）。
- 用 AI 自動化滑鼠點擊在這個視窗上極不穩定（座標一直對不準正確的按鈕），來回試了十幾次才放棄自動化，改成「使用者自己操作，AI 在旁邊看截圖判讀」。
- 樹狀結構很深：`SWindow → SVerticalBox →（先走錯一條叫 SWindowTitleBar 的死路，那只是視窗標題列）→ 退回來 → SGlobalPlayWorldActions → SPIEViewport → SGameLayerManager → SDPIScaler →（再走錯一次）→ 終於找到真正裝畫面內容的 SOverlay`。
- 底下一排 `SConstraintCanvas`，每一個對應一個 `AddToViewport()` 加進去的 widget。展開了快 12 個，前面 9 個全部是「Collapsed」的開發面板，第一個是正常運作的 `WBP_PlayerHUD`。
- **第 12 個**：Visibility 顯示 `SelfHitTestInvisible`（不是 Collapsed，代表「應該要顯示」），但底下的子節點只是一個空的 `SSpacer`——不是 Border，不是任何畫面內容。

**這就是兇手。**

回頭查引擎原始碼 `UserWidget.cpp:1203`：

```cpp
TSharedRef<SWidget> UserRootWidget = WidgetTree->RootWidget
    ? WidgetTree->RootWidget->TakeWidget()
    : TSharedRef<SWidget>(SNew(SSpacer));   // ← 就是這個
```

`UWidget::TakeWidget_Private()` 的真實執行順序是：**先**呼叫 `RebuildWidget()`（讀取 `WidgetTree->RootWidget`，這時候是 null，回退成空 SSpacer，結果定型），**事後**才呼叫 `NativeConstruct()`。而整個專案的 UI 慣例，寫在 CLAUDE.md 裡，明明白白教大家「在 `NativeConstruct()` 裡呼叫 `BuildLayout()`」——這個慣例本身，從根上就是錯的。任何照這個慣例寫的 widget，第一次 `AddToViewport()` 必定拿到空畫面，而且因為 `MyWidget` 一旦被快取就不會重建，這個空畫面是**永久性**的。

**修法**：把 `BuildLayout()` 從 `NativeConstruct()` 搬到 `NativeOnInitialized()`——這個會在 `CreateWidget()` 當下就執行，早於 `AddToViewport()`，`WidgetTree->RootWidget` 會在第一次 `RebuildWidget()` 檢查它之前就準備好。改完重 build，實機截圖——

**標題畫面終於出現了。**

### 戰損統計

- 從「回報完成」到「實際修好」：跨了兩天、十幾輪「改程式碼→重 build→重開 Editor→重測」循環
- 排除了 10 個合理但錯誤的假設
- 用掉了一個假修復的考古發現（commit `17dc0a0`）
- Widget Reflector 操作本身的學習曲線，可能比 bug 本身還陡

### 教訓（寫給未來的自己）

1. **`Build 0 錯誤 0 警告` 跟「畫面真的對」是兩件事**，中間隔著一整層 UMG 生命週期。Log 印出「widget created, added to viewport」只證明程式碼跑到那一行，不證明任何東西被畫出來。
2. **commit 訊息聲稱修好了，不代表真的修好了**——尤其如果那個 commit 用了一個查證後根本不存在的 API，連編譯都過不了，整個「修復」可能從頭到尾沒被驗證過。
3. **`NativeConstruct()` 不是建構 WidgetTree 安全的地方**——這是這次最重要的技術發現，已經寫進 CLAUDE.md 待更新的清單。正確位置是 `NativeOnInitialized()`。
4. **這個 bug 不只 GameFlowWidget 中獎**——專案裡所有照同一套慣例寫的 widget（Inventory/Equipment/Settings 等）都可能是空的，只是因為預設 Collapsed 沒人發現。已經開了背景任務要逐一稽核。
5. 當程式碼層面的假設全部排除完還是沒答案時，**該換工具，不是換假設**。Widget Reflector 比再猜 10 個假設快得多——但早點承認「我猜不出來了，需要直接看渲染樹」會省下大把時間。
