# 開發血汗錄

> 記錄那些「程式碼看起來都對，但畫面就是不對」的修羅場。技術細節在 commit/issues.md 裡都有，這份檔案記的是過程——花了多久、走錯幾次、最後怎麼抓到的，給未來的人（不管是人類還是 AI）一個警世故事。

---

## 第 1 案：標題畫面的幽靈（2026-06-20 ~ 2026-06-21）

**症狀**：GameFlowWidget 整套重做完，`Build 0 錯誤 0 警告`，commit 完，回報「完成」。使用者按下 Play——直接跳過標題畫面，進到遊戲場景。HP/MP 條都在，就是沒有那層「SkillCreator」標題。

**第一輪診斷（自己跑 `-game`，純看 log）**：

跑了三次 `UnrealEditor-Cmd.exe -game`，log 都乖乖印著：

```
LogTemp: SkillCreatorGameMode: no pending world/character — showing GameFlowWidget
LogTemp: SkillCreatorGameMode: GameFlowWidget created, added to viewport, game paused
```

沒有任何 Error。看起來邏輯都對，於是先去修了個真的存在但無關的小 bug（`GlobalDefaultGameMode` ini 路徑寫錯 `ASkillCreatorGameMode` 多了個 `A`），回報「應該沒問題了」。

**使用者**：「你還是失敗了，進入遊戲依然沒有看到標題畫面。」

——這時候才發現，光看 log 完全不夠。Log 只能證明「程式碼執行到那一行沒有 crash」，不能證明「畫面上真的畫出東西」。

**第二輪診斷（追到變成 stale binary 烏龍）**：

發現使用者測的 `Binaries/Win64/SkillCreatorUE5.exe` 是 4 天前編的，跟 Editor target 是兩個完全獨立的編譯產物，互不更新。修正了，重 build 兩個 target。使用者再測——

**還是沒有畫面。**

**第三輪診斷（瘋狗式排除法，大概燒了 2 小時）**：

接下來進入了一段非常痛苦的「拿掉一個變因、重 build、重測」循環，每輪大概 3~4 分鐘：

| # | 排除的假設 | 結果 |
|---|-----------|------|
| 1 | Widget 尺寸算成 0×0 | 量出來 `AbsSize=(900,650)` 滿版，不是 |
| 2 | `SetAnchorsInViewport`/`SetAlignmentInViewport` 設定有問題 | 拿掉照樣壞，UE5.7 預設本來就是滿版拉伸 |
| 3 | `SetPause`/`SetInputMode`/`SetShowMouseCursor` 副作用 | 拿掉照樣壞 |
| 4 | 跟 `AHUD::BeginPlay()` 的執行順序有關 | 延後 30 個 tick 建立，照樣壞 |
| 5 | Border 沒設 Brush 資源不會畫 | 換成最簡單的純紅色 Border + 文字，照樣壞 |
| 6 | 連 Border 都不要，只留一個黃色文字當 root | **還是完全沒畫出來** |
| 7 | 是不是「誰呼叫 `CreateWidget`」這個 context 的問題 | 搬到跟會渲染的 HUDWidget 同一個 context 建立，照樣壞 |
| 8 | `UCLASS()` 缺 `BlueprintType, Blueprintable` 標記 | 補上，照樣壞 |

每一項都合理懷疑、每一項都查了 UE5.7 引擎原始碼驗證、每一項都實機重測——**全部沒用**。

中途還挖出一個離題的小發現：git log 裡有個 6/20 的 commit，標題正是「修復 GameFlowWidget 在 PIE 中永遠不可見的問題」，用了一個叫 `SetOffsetsInViewport()` 的 API。一查引擎原始碼——**這個方法在 UE 5.7 根本不存在**。後來有個 session 為了修編譯錯誤把那行刪了，但沒人意識到這可能等於把「修復」本身也刪掉了。也沒人重新看過畫面。這個 bug 大概已經潛伏了一天多，從頭到尾沒人真的用眼睛確認過。

**轉折點：Widget Reflector**

純程式碼/log 排查走到死路，唯一剩下的工具是 Slate 的 Widget Reflector——直接看引擎內部真正畫出來的 widget 樹。

這段也不輕鬆：

- 一開始選了「Standalone Game」play 模式——這是**獨立 process**，Widget Reflector 在 Editor process 裡，完全看不到另一個 process 的東西。換成「New Editor Window (PIE)」才解決。
- Widget Reflector 視窗一開始太小，操作幾次後不小心被拖到背景去（使用者一度以為「視窗消失了」）。
- 用 AI 自動化滑鼠點擊在這個視窗上極不穩定（座標一直對不準正確的按鈕），來回試了十幾次才放棄自動化，改成「使用者自己操作，AI 在旁邊看截圖判讀」。
- 樹狀結構很深：`SWindow → SVerticalBox →（先走錯一條叫 SWindowTitleBar 的死路，那只是視窗標題列）→ 退回來 → SGlobalPlayWorldActions → SPIEViewport → SGameLayerManager → SDPIScaler →（再走錯一次）→ 終於找到真正裝畫面內容的 SOverlay`。
- 底下一排 `SConstraintCanvas`，每一個對應一個 `AddToViewport()` 加進去的 widget。展開了快 12 個，前面 9 個全部是「Collapsed」的開發面板，第一個是正常運作的 `WBP_PlayerHUD`。
- **第 12 個**：Visibility 顯示 `SelfHitTestInvisible`（不是 Collapsed，代表「應該要顯示」），但底下的子節點只是一個空的 `SSpacer`——不是 Border，不是任何畫面內容。

**這就是兇手。**

回頭查引擎原始碼 `UserWidget.cpp:1203`：

```cpp
TSharedRef<SWidget> UserRootWidget = WidgetTree->RootWidget
    ? WidgetTree->RootWidget->TakeWidget()
    : TSharedRef<SWidget>(SNew(SSpacer));   // ← 就是這個
```

`UWidget::TakeWidget_Private()` 的真實執行順序是：**先**呼叫 `RebuildWidget()`（讀取 `WidgetTree->RootWidget`，這時候是 null，回退成空 SSpacer，結果定型），**事後**才呼叫 `NativeConstruct()`。而整個專案的 UI 慣例，寫在 CLAUDE.md 裡，明明白白教大家「在 `NativeConstruct()` 裡呼叫 `BuildLayout()`」——這個慣例本身，從根上就是錯的。任何照這個慣例寫的 widget，第一次 `AddToViewport()` 必定拿到空畫面，而且因為 `MyWidget` 一旦被快取就不會重建，這個空畫面是**永久性**的。

**修法**：把 `BuildLayout()` 從 `NativeConstruct()` 搬到 `NativeOnInitialized()`——這個會在 `CreateWidget()` 當下就執行，早於 `AddToViewport()`，`WidgetTree->RootWidget` 會在第一次 `RebuildWidget()` 檢查它之前就準備好。改完重 build，實機截圖——

**標題畫面終於出現了。**

### 戰損統計

- 從「回報完成」到「實際修好」：跨了兩天、十幾輪「改程式碼→重 build→重開 Editor→重測」循環
- 排除了 10 個合理但錯誤的假設
- 用掉了一個假修復的考古發現（commit `17dc0a0`）
- Widget Reflector 操作本身的學習曲線，可能比 bug 本身還陡

### 教訓（寫給未來的自己）

1. **`Build 0 錯誤 0 警告` 跟「畫面真的對」是兩件事**，中間隔著一整層 UMG 生命週期。Log 印出「widget created, added to viewport」只證明程式碼跑到那一行，不證明任何東西被畫出來。
2. **commit 訊息聲稱修好了，不代表真的修好了**——尤其如果那個 commit 用了一個查證後根本不存在的 API，連編譯都過不了，整個「修復」可能從頭到尾沒被驗證過。
3. **`NativeConstruct()` 不是建構 WidgetTree 安全的地方**——這是這次最重要的技術發現，已經寫進 CLAUDE.md 待更新的清單。正確位置是 `NativeOnInitialized()`。
4. **這個 bug 不只 GameFlowWidget 中獎**——專案裡所有照同一套慣例寫的 widget（Inventory/Equipment/Settings 等）都可能是空的，只是因為預設 Collapsed 沒人發現。✅ **2026-06-24 稽核完成**：Inventory/Equipment/Settings/ShapeMenu/SpellGroup/Stats/InputSettings/SpellList/DebugPaint/BlockEditor 全部確認已使用 `NativeOnInitialized()`，風險消除。
5. 當程式碼層面的假設全部排除完還是沒答案時，**該換工具，不是換假設**。Widget Reflector 比再猜 10 個假設快得多——但早點承認「我猜不出來了，需要直接看渲染樹」會省下大把時間。

## 第 2 案：陰魂不散的格子地面（2026-06-23，未解決，留給下一個 session）

**症狀**：使用者反覆回報「地面是棋盤格紋」，這個症狀至少在這個專案裡出現過 3 次，每次都「修好了」，每次都復發。這次第三度復發，使用者實機截圖確認：新角色+新世界、封裝版 exe（不是 PIE），整片地面到目視可見的最遠處全是格子狀（不是只有一格、不是只有出生點附近）。

**這次的排查跟前兩次不一樣的地方**：前兩次的「修法」（`TileMaterialRegistry::GetSurface()` 改用 `LoadSynchronous()`、清掉 `MainMap.umap` 的 GameMode override）都是對的、都確實解決了當時抓到的那個根因，但**顯然還有另一個獨立的根因沒被抓到**，這次決定不再猜「會不會是同一個老問題」，改成從頭逐層用診斷 log + 讀 plugin 原始碼重新驗證每一個環節。

**已經排除、確認沒問題的環節（下一個 session 不要再往這幾個方向查）**：

1. ✅ **`TileMaterialRegistry->GetSurface(EMaterialType::Grass)` 確實回傳非 null**——封裝版自己的 log（`Saved/Logs/SkillCreatorUE5.log`）裡 `AVoxelWorldActor::BeginPlay slot 3 Mat=M_Tile_Grass` 清楚印出來，不是 `NULL(from registry)`。`LoadSynchronous()` 修法本身依然有效。
2. ✅ **`RMCMesh->SetupMaterialSlot(3, ..., M_Tile_Grass)` 確實被呼叫**——同一行 log 證明。
3. ✅ **PolyGroup 數值直接等於 MaterialID**——`GreedyMesher.cpp:185 PolyGroupBuilder.Add(Mat)`，`Mat` 就是 tile 的 `EMaterialType` byte 值，沒有經過任何轉換。
4. ✅ **RealtimeMeshComponent plugin 的 PolyGroup→MaterialSlot 映射是直接 1:1**——讀了 plugin 原始碼 `RealtimeMeshSectionConfig.h:15`：`FRealtimeMeshSectionConfig(int32 InMaterialSlot) : MaterialSlot(InMaterialSlot)`，`DefaultPolyGroupSectionHandler(PolyGroupIndex)` 直接 `return FRealtimeMeshSectionConfig(PolyGroupIndex)`。`CreateSectionGroup()` 預設 `bShouldAutoCreateSectionsForPolyGroups=true`，理論上每個 PolyGroup 數值都會自動建立對應同編號的 MaterialSlot 區段。
5. ✅ **`M_Tile_Grass` 材質本身不可能自己產生格紋**——用 Python（`UnrealEditor-Cmd.exe -run=pythonscript`）直接內省這個材質：`vec params/scalar params/texture params` 全部是空陣列（不是 Material Instance、沒有任何貼圖），`BaseColor` 接的是 `MaterialExpressionConstant3Vector`（純常數色，數學上是均勻單色，不可能自己變成棋盤格）。
6. ✅ **不是因為 cook 漏掉資產**——直接檢查 `Saved/Cooked/Windows/.../Content/VoxelMaterials/M_Tile_Grass.uasset` 確實存在，`DA_TileMaterialRegistry.uasset` 也存在，封裝過程沒有漏掉這些資源。
7. ✅ **`Validate()` 沒有被任何地方呼叫**，不可能因為驗證失敗導致整個 Registry 被停用。

**有意思的旁支發現（但確認跟這個 bug 無關）**：`/Game/M_Voxel.M_Voxel`（fallback 材質）的 `BaseColor` 接的是 `MaterialExpressionVertexColor`，不是常數色——而 `GreedyMesher.cpp:154` 確實會把 `MaterialID`/`Variant` 編碼進 vertex color（`FColor Col(Mat, VariantMask[...], 0, 255)`，註解寫「H-4：要在材質端讀 VertexColor.G 做基色微調，這部分屬美術資產編輯，待手動」）。**這代表如果有任何地方真的 fallback 到 `M_Voxel`，視覺上就會是「材質ID/變異值被解讀成顏色」，很可能就長得像格紋/雜訊圖案**——這是目前最接近「能解釋視覺symptom」的理論，但反覆確認 slot 3 設定的是 `M_Tile_Grass` 不是 `M_Voxel`（見上面第 1、2 點），所以「為什麼實際渲染出來的畫面看起來像是套用了 `M_Voxel`」這個問題本身還沒解開。

**下一個 session 該往哪裡查**（按優先順序）：

1. **先確認是不是封裝版專屬**：同一個材質設定在 Editor PIE 測試會不會也出現格紋。如果 PIE 正常、只有封裝版有問題，方向應該轉去查 cook/shader 編譯（封裝平台的 shader 有沒有編譯失敗，UE5 對編譯失敗的材質有內建的棋盤格 fallback 外觀，這跟症狀描述高度吻合，但這次沒查到 shader compile error 的直接證據，cook log 顯示 `ShadersCompiled=0`，意義不明確，值得用 `-LogCmds="LogShaderCompilers Verbose"` 之類更詳細的 shader log 重跑一次封裝再確認）。
2. **棋盤格範圍有沒有邊界**：玩家走到很遠的地方，棋盤格地面會不會在某個距離後變成正常顏色？如果有邊界，代表只是「出生點附近某個特定 Mega-Chunk 的渲染」出問題，範圍會大幅縮小；如果無邊界，才是真正全域的材質/渲染管線問題。
3. 確認 `URealtimeMeshSimple::CreateSectionGroup()` 實際執行時，是不是真的有依照 PolyGroup 自動建立分段——理論上讀 plugin 原始碼是這樣設計，但沒有用 debugger 或額外 log 直接驗證「執行期真的照這個邏輯跑」，純讀程式碼跟「這段程式碼真的被以這個方式呼叫到」中間還有一層沒驗證到。

**這次順手抓到、已修復、但跟上面這個 bug 無關的東西**：
- `UCraftingPanelWidget.cpp`（2026-06-23 新寫）忘了套用既有的 `SetBrushColor()→SetBrush(MakeSolid)` 修法，導致面板顯示成不正確的鮮綠色色塊——這帶出第二個發現：Bug H-4 的 commit 訊息聲稱「全專案所有 SetBrushColor() 呼叫都改掉了」，但 grep 全專案後發現還有 18+ 個檔案沒被修到，H-4 那次的修復範圍被誇大了，沒有真的「全專案」。
- `EMaterialType::Fixture`（本次 session 新增的列舉值）沒有排除在 `GreedyMesher::IsOpaque()` 之外，會被當一般材質渲染但 Registry 沒有對應資料，範圍很小（只影響寶箱/工作臺腳下那一格），跟格子地面這個全域症狀無關，但既然查到了就順手修掉。

### 教訓

1. **同一個視覺症狀，可能來自完全不同的根因，即使外觀一模一樣**。這次第三度復發的格子地面，前兩次的修法都還是對的（沒有迴歸），但顯然從一開始就有第二個獨立問題沒被抓到，只是被前兩個根因「蓋住」了，前兩個修完之後這個才浮出水面。
2. **純讀程式碼可以排除很多假設，但排除到底也不一定能找到答案**——這次系統性地排除了 7 個環節，每個都用 log 或原始碼證據確認過，但仍然沒有解開「為什麼畫面看起來不對」。跟第 1 案一樣，純 log/原始碼排查到死路時，下一步該換工具（Editor PIE 對照測試、更詳細的 shader log、甚至 RenderDoc 截幀），不是繼續猜假設。
3. **commit/修復記錄裡的「全專案」「已修復」字眼要保持懷疑**——這次親身驗證了 Bug H-4 的「全專案」說法不實，這跟第 1 案教訓 2「commit 訊息聲稱修好了不代表真的修好了」是同一個教訓的第二次驗證。

## 第 3 案：輻射地板靜默失效（2026-06-26）

**症狀**：`EContactEffect::Radioactive` 材質不會對玩家施加任何元素 Aura。玩家站在輻射地板上沒有任何 debuff，且沒有任何 log 輸出，程式跑得毫無異常。

**根因**：

`ApplyEnvironmentalDamage()` 裡的 switch 在 Phase 4 實作時，手動列出了 5 個 EContactEffect 值的映射：Burning, Wet, Poison, Electric, Frozen → 各自對應 ESkillElementType。Radioactive（第 6 個）被遺漏了，落入 `default: break;`，整個什麼都沒做。

```cpp
// 修前（漏掉 Radioactive）：
switch (CurData.ContactEffect)
{
    case EContactEffect::Burning:  ContactElem = ESkillElementType::Fire;    break;
    case EContactEffect::Wet:      ContactElem = ESkillElementType::Water;   break;
    case EContactEffect::Poison:   ContactElem = ESkillElementType::Poison;  break;
    case EContactEffect::Electric: ContactElem = ESkillElementType::Thunder; break;
    case EContactEffect::Frozen:   ContactElem = ESkillElementType::Ice;     break;
    default: break;  // ← Radioactive 在這裡靜默消失
}
```

**為什麼沒被馬上發現**：
1. `ESkillElementType` 裡沒有 Radioactive（蒼究五行體系無此元素），設計者知道「沒有直接映射」，所以直覺上以為 `default: break;` 是對的——但正確做法應該是找最接近的元素（Poison），而不是直接放棄整個效果。
2. UE5 的 `switch/default` 不會對 unhandled enum value 產生任何 warning，編譯完全乾淨。
3. 程式沒有 crash，Radioactive 材質本身工作正常（DangerFlags bit3 還在），只有 AuraComp 效果靜默遺失。

**怎麼被找到的**：

預檢腳本新增了 Check 13p（`EContactEffect → ApplyEnvironmentalDamage 映射完整性`）：讀取 `MaterialRegistry.h` 裡的 `EContactEffect` 所有 enum 值，逐一確認 `ASkillCreatorCharacter.cpp` 裡是否出現 `EContactEffect::X` 字串。執行時直接回報 **FAIL**，修法：補 `case EContactEffect::Radioactive: ContactElem = ESkillElementType::Poison; break;`（輻射→毒性是遊戲慣例，蒼究體系無輻射元素，Poison 是語義上最近的 fallback）。

**所花時間**：0 分鐘。預檢直接抓到，不需要實機重測。

### 教訓

1. **手動列舉 enum 值的 switch 天生脆弱**——每次 enum 新增值都可能讓現有 switch 靜默擴展 `default` 範圍。有新 enum 值時，要主動搜尋全專案所有相關 switch 並確認每個值都被處理。
2. **`default: break;` 是靜默的黑洞**——不是 crash，不是 warning，不是任何輸出。這種 bug 在純程式碼審查或靜態分析下幾乎不可能被人眼識別。
3. **預檢腳本的「枚舉完整性」守衛是這種 bug 唯一的自動化屏障**——把 enum↔switch 完整性檢查加進預檢腳本，是投資報酬率最高的預防手段。
4. **沒有 ESkillElementType 直接對應的 ContactEffect 值，不是「不映射」的理由**——應該選最接近的語義 fallback 並在 case 裡明確標注原因，而不是靜默落入 default。
